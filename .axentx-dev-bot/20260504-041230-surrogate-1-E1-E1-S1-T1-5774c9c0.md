## Unified “Ingest → Process → Train → Publish” Pipeline  
Below is a **single, end‑to‑end GitHub‑Actions solution** that stitches together the strongest ideas from the four draft templates you shared:

| Stage | What the original draft did well | What we keep / improve | Why it matters |
|-------|----------------------------------|------------------------|----------------|
| **Data Ingest** | Streaming, schema validation, batch‑size control, artifact upload. | • Re‑usable workflow (`ingest.yml`). <br>• Explicit `schema` object (JSON‑Schema) for validation. <br>• Automatic retry with `continue-on-error` + `max-attempts`. | Guarantees clean, deduplicated raw data that downstream jobs can rely on. |
| **Parallel Processing** | Deterministic sharding, configurable workers, artifact per‑shard. | • Built‑in **matrix** strategy to spin up the exact number of shards you ask for (no manual “launch‑a‑run‑per‑shard”). <br>• Centralised deduplication step after all shards finish. | Removes human error, scales automatically, and produces a single merged output. |
| **Model Training** | Hyper‑parameter flexibility, checkpointing, metric logging. | • Runs **only after** the merged dataset is ready. <br>• Uses a Docker container with GPU support (if you have self‑hosted runners). <br>• Saves checkpoints as artifacts *and* pushes them to a model‑registry (HF, S3, GCS). | Guarantees reproducible training and makes the model instantly reusable. |
| **Dataset / Model Publish** | Platform‑agnostic upload (HF, S3, GCS). | • Single reusable `publish.yml` that can upload **both** processed datasets *and* trained models. <br>• Handles authentication via repository‑secrets (`HF_TOKEN`, `AWS_ACCESS_KEY_ID`, …). | One place to manage credentials and versioning, avoiding duplicated code. |

## 1. Repository Layout

```
.github/
└─ workflows/
   ├─ ingest.yml            # reusable ingest workflow
   ├─ parallel.yml          # reusable parallel‑processing workflow
   ├─ train.yml             # reusable model‑training workflow
   ├─ publish.yml           # reusable upload/publish workflow
   └─ orchestrator.yml      # **master** pipeline that calls the four above
templates/
└─ scripts/
   ├─ validate_schema.py   # tiny helper used by ingest
   ├─ shard_processor.py   # deterministic sharding logic
   └─ train.py              # generic training entry‑point
```

All scripts are deliberately tiny (≈ 30 LOC each) and can be swapped out for your own code without touching the CI definition.

## 2. Re‑usable Workflow Files  

### 2.1 `ingest.yml`

```yaml
# .github/workflows/ingest.yml
name: Ingest
on:
  workflow_call:
    inputs:
      source_url:
        required: true
        type: string
      target_path:
        required: true
        type: string
      schema:
        required: false
        type: string   # JSON‑Schema string (or path to file)
      batch_size:
        required: false
        type: number
        default: 1000
jobs:
  ingest:
    runs-on: ubuntu-latest
    timeout-minutes: 120
    steps:
      - uses: actions/checkout@v4

      - name: Install utilities
        run: |
          python -m pip install --quiet pandas jsonschema tqdm

      - name: Download source
        env:
          SRC: ${{ inputs.source_url }}
        run: |
          mkdir -p ${{ inputs.target_path }}
          curl -L "$SRC" -o raw_data.tmp
          # If the source is a CSV/JSON stream we pipe through tqdm for progress
          # (replace with your own downloader if needed)

      - name: Validate & Normalize (optional)
        if: inputs.schema != ''
        run: |
          python scripts/validate_schema.py \
            --input raw_data.tmp \
            --schema "${{ inputs.schema }}" \
            --output ${{ inputs.target_path }}/normalized.parquet \
            --batch-size ${{ inputs.batch_size }}

      - name: Upload raw artifact
        uses: actions/upload-artifact@v4
        with:
          name: raw-data
          path: ${{ inputs.target_path
