from datetime import datetime, timedelta
from app.models.budget_rules import BudgetRules
from app.utils.email_service import send_email

def check_budget_and_alert():
    budget_rules = BudgetRules.get_current_rules()
    today = datetime.now()
    one_week_ago = today - timedelta(days=7)

    for budget in budget_rules:
        total_spend = budget.get_total_spend(one_week_ago, today)
        if total_spend >= (budget.budget * 0.8):
            days_left = (datetime(today.year, today.month + 1, 1) - today).days
            remaining_budget = budget.budget - total_spend
            forecast = remaining_budget / days_left
            alert_message = f"Your budget is {total_spend} out of {budget.budget}. You have {days_left} days left with a forecast of {forecast} per day."
            send_email(budget.user.email, "Budget Alert", alert_message)

Short answer: The proposed change is a workable first step toward an automated budget alert system, but it needs concrete implementations and safeguards to be production-ready.

Assessment by parts
- check_budget_and_alert logic
  - Pros: Centralizes budget checks, uses a weekly window, sends alerts when 80% of budget is reached, includes forecast derived from days left. This aligns with typical budget alert use cases.[1][2]
  - Cons / gaps:
    - The function iterates budget_rules without filtering for a current month or user scope; if BudgetRules.get_current_rules() returns all records, alerts could trigger for past periods or inactive budgets. You may want to scope by month, user, or active flag.[3]
    - total_spend calculation relies on BudgetRules.get_total_spend, which is a stub in your snippet. You need a concrete query (e.g., sum of expenses within a date range joined to the budget) to avoid runtime errors.[3]
    - The days_left calculation uses a simplistic approach (first day of next month). It incorrectly uses month + 1 without handling December wrap to January of next year, and it assumes a fixed month boundary rather than calendar-driven. This will produce wrong results near month ends. Consider computing days_left as (start_of_next_month - today).days.[3]
    - The forecast divides remaining_budget by days_left; if days_left <= 0, division by zero occurs. Add a guard for non-positive days_left.[2]
    - Email sending relies on send_email; ensure retries, backoff, and error handling if SMTP/server is unavailable.[4][2]
- budget_rules.py get_current_rules
  - Pros: A classmethod stub is convenient; using Peewee for ORM fits light-weight DBs.[5][6]
  - Cons / gaps:
    - get_current_rules returns all records; you’ll likely want to filter to the current time frame (e.g., current month/year) or a per-user active budget rule.[3]
    - get_total_spend is a stub; you must implement a query that sums spend from a expenses table (joined by user and budget) over start_date to end_date.[3]
- data model alignment
  - The BudgetRules model includes user, budget, and currency; ensure currency handling is consistent across calculations and that budget is a numeric type (DecimalField is good).[5]
  - If you expect multiple budgets per user (categories, projects), consider a separate Budget and Expense model with foreign keys to User and BudgetRule to simplify queries.[3]

Concrete recommendations to move toward workable step forward
