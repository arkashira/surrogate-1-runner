## Final Design – “Hybrid” Cost Analyzer

The two proposals share the same goal – generate a list of actionable cost‑saving recommendations – but they differ in the way they decide *which* resources are worth acting on:

| Proposal | Decision rule | What it flags |
|----------|---------------|---------------|
| **Candidate 1** | *Absolute* cost > $1 000 | High‑cost resources |
| **Candidate 2** | *Relative* cost < 10 % of the resource’s average | Under‑utilised resources |

Both rules are useful in a real‑world environment.  The best solution is to expose **both** thresholds as configurable parameters, let the caller decide which rule(s) to apply, and return a single, sorted list of recommendations.

Below is a single, production‑ready implementation that:

1. **Aggregates** historical cost records by resource ID.
2. **Computes** both the total cost and the average daily cost for each resource.
3. **Applies** the two configurable thresholds (high‑cost *and* under‑utilised) – a resource can be flagged by either rule.
4. **Estimates** potential savings:
   * For high‑cost resources – 30 % of the total cost (as in Candidate 1).
   * For under‑utilised resources – the full average cost (i.e. shutting it down would save that amount).
5. **Merges** the two recommendation lists, removes duplicates, and sorts by potential savings (descending).
6. **Exposes** a clean API and a small test suite that validates the behaviour.

> **Why this is better than either candidate alone?**  
> *It gives the caller full control over the policy, avoids hard‑coding a single rule, and keeps the code DRY by re‑using the same aggregation logic.*
