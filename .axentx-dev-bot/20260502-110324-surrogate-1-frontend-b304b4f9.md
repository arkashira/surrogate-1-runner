**Diagnosis and Problem Statement:**

The frontend application relies heavily on authenticated API calls to the Hugging Face (HF) API for every session, leading to intermittent 429 errors (too many requests) and slow cold starts. The absence of a persisted CDN manifest cache means that each visit re-enumerates the repository tree instead of using a deterministic date-folder file list. The lack of retry/backoff and CDN fallback mechanisms causes transient network errors to break data loads. Furthermore, the UI does not clearly indicate the data source (CDN vs. API) or staleness, resulting in poor transparency for users. Lastly, there is no TTL (time to live) or refresh strategy for the manifest, allowing stale file lists to linger without a clear invalidation path.

**Proposed Solution:**

To address these issues, the proposal suggests creating a new file, `/opt/axentx/surrogate-1/frontend/src/lib/data/cdnManifest.ts`, which will serve as a deterministic CDN manifest loader. This loader will:

1. **Fetch Manifest:** Load a manifest produced by a Mac orchestrator (which makes a single HF API call to list the repository tree and saves it to a JSON file). The frontend will consume this manifest and fetch parquet files via CDN URLs only.
2. **Add Retry/Backoff and CDN Fallback:** Implement typed fetches with exponential backoff and CDN fallback to handle transient network errors and avoid breaking data loads.
3. **Expose Source/Staleness Metadata:** Make the data source (CDN vs. API) and staleness visible to the UI, enhancing transparency for users.
4. **Implement TTL for Manifest Refresh:** Set a TTL of 15 minutes for the manifest refresh to ensure that stale manifests are periodically updated.

**Implementation Details:**

- The `cdnManifest.ts` file will contain functions for loading the CDN manifest, fetching parquet files from the CDN, and listing date folders (though the latter is discouraged in favor of pre-generated manifests).
- The `loadCdnManifest` function will fetch the manifest, parse it, and return a `CdnManifest` object that includes metadata such as the date folder, generation time, expiration time, and a list of files with their paths, sizes, and CDN URLs.
- The `fetchCdnParquet` function will stream a parquet file from the CDN, allowing for the parsing and projection of data at parse time.
- The `listDateFolders` function, though not recommended for frequent use due to rate limiting, can be used to get the latest available date folders from a top-level listing.
