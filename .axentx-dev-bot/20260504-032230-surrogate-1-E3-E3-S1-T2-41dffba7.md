## Unified PasteResolver design  

The two drafts each contain useful pieces, but they also conflict:

| Issue | Draft 1 | Draft 2 | Resolution |
|-------|---------|---------|------------|
| **Constructor** | Takes a `java.awt.datatransfer.Clipboard` (good for unit‑testing) | No‑arg constructor (harder to mock) | Keep the injectable `Clipboard` **and** add a no‑arg convenience constructor that uses the system clipboard. |
| **Clipboard read** | Uses `clipboard.getData(DataFlavor.stringFlavor)` (correct) | – | Keep this implementation. |
| **AppleScript execution** | Low‑level `ProcessBuilder` that writes the script to stdin (works but the stream handling is fragile) | One‑liner `osascript -e …` (simpler) | Provide a single, well‑tested `executeAppleScript(String script)` that can be called with any script. Internally it will use `ProcessBuilder` with `-e` arguments (no need to pipe). |
| **Return type for AppleScript** | `void` (throws on failure) | `boolean` (true = success) | Expose both: a public `boolean runAppleScript(String script)` that returns success, and a private helper `void executeAppleScript(String script)` that throws – this lets callers choose the style they prefer. |
| **Unit tests** | Uses Mockito to stub `Clipboard.getData` – precise and fast. | Tests the AppleScript path with a real process (slow, flaky). | Keep the **clipboard** test (mocked) and add a **light‑weight AppleScript test** that only runs when the OS is macOS and `osascript` is present. The test will be disabled on other platforms. |
| **Error handling** | Wraps any exception in `RuntimeException`. | Swallows exception and returns `false`. | Log the exception (using `java.util.logging`) and re‑throw a `RuntimeException` for the “throwing” API, while the boolean API returns `false`. |

Below is the **final, production‑ready implementation** together with a **complete test suite** that merges the strongest ideas from both proposals.
