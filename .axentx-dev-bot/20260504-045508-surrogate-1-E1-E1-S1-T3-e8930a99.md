### How we resolved the contradictions  

| Conflict | Resolution |
|----------|------------|
| **`onChange` vs `onFilterChange`** | The final component accepts **both** props. If only one is supplied we use it; if both are supplied we call both (so existing code keeps working). |
| **Filter object shape** (`priceMin`/`priceMax` vs `price.{min,max}`) | The final filter object contains **both** representations for backward compatibility: `priceMin`/`priceMax` (numbers or `null`) **and** a nested `price` object (`{ min, max }`). Consumers can pick whichever they prefer. |
| **Spec values as free‑text vs dropdown options** | The UI renders a **text input** by default. If a spec definition includes an `options` array, we render a `<select>` instead. This gives the flexibility of Candidate 2 while keeping the generic behaviour of Candidate 1. |
| **Prop‑type shape** (`categories` as strings vs objects) | The component normalises the `categories` prop: it accepts either an array of strings **or** an array of `{ id, name }` objects. Internally we always work with `{ id, label }` objects. |
| **State initialisation** (empty string vs `null`) | We store the raw UI values as strings (`""` for “no selection”). When we build the filter object we coerce to `null` or `Number` as appropriate, matching both earlier designs. |
