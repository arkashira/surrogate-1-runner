## Unified, Production‑Ready Routing Test Suite  

Below is a single **pytest**‑based test module that merges the strongest ideas from the two proposals, resolves their contradictions, and adds a few extra safety nets.  

* **Why pytest?**  
  * The original project already ships a `pytest`‑style test runner (see the `TestClient` import in the first draft).  
  * `pytest` fixtures give us clean setup/teardown of a temporary routing file **and** automatic cleanup of environment variables.  

* **What we keep from each draft?**  
  * **Temporary YAML routing file** – guarantees the tests never touch the real production config.  
  * **Deterministic routing check** – identical payloads must always resolve to the same agent.  
  * **Hot‑reload verification** – modify the file on‑disk, wait a tick, then force a reload; the new mapping must be used.  
  * **Invalid‑route handling** – a 400 response with a helpful error message.  
  * **Schema validation** – we now also assert that the routing table is rejected when it contains an unknown hash function (the “invalid routing table” idea from the second draft).  

* **How contradictions are solved**  
  * The second draft used a **unit‑test‑only** approach with mocked file I/O, while the first draft exercised the **real FastAPI endpoint**.  
  * We keep the **real endpoint** (so the whole stack – file watcher → router → HTTP layer – is exercised) **and** add a **pure‑Python unit test** that validates the `RoutingTable` constructor’s schema logic.  
  * Both styles coexist in the same file, each under its own `pytest` marker, so they can be run together or separately.

    # ------------------------------------------------------------------
