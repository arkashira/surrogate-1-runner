## Unified, Production‑Ready Paste‑Handling Stack  

Below is a **single, self‑contained implementation** that merges the strongest ideas from the two proposals you posted:

| What we keep | Why it is the best |
|--------------|-------------------|
| **`AxPasteHandler`** that first tries the **real macOS Accessibility (AX) API** (via `pyobjc`) and, if that is unavailable or fails, falls back to a **tiny AppleScript** that copies the text to the clipboard and sends a Cmd‑V keystroke. | Gives the most direct, UI‑independent paste when the AX API works, but still works on machines without `pyobjc` (e.g. CI runners, older macOS) by using the proven AppleScript shim. |
| **`CgEventPasteHandler`** (unchanged) – a pure‑Quartz “type Cmd‑V” fallback that works for any app that honours the system pasteboard. | Guarantees a working path even when the focused element cannot be set via AX. |
| **`AppleScriptPasteHandler`** (unchanged) – a pure‑AppleScript fallback that also sets the clipboard before typing Cmd‑V. | Provides a second, independent way to type Cmd‑V in case the low‑level CGEvent code crashes or is blocked by sandboxing. |
| **`PasteDispatcher`** that iterates over a **list of handlers** in priority order, logs each attempt, and returns as soon as one succeeds. | Simple, extensible, and defensive – a new handler can be added by just appending to the list. |
| **Robust logging** (debug, info, warning, error) and **exception safety** around every handler. | Makes troubleshooting painless in production. |
| **All strings are safely escaped** before being injected into AppleScript. | Prevents malformed scripts when the text contains quotes or new‑lines. |
