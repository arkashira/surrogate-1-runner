| Candidate Strength | How we keep it | Why it matters |
|--------------------|----------------|----------------|
| **Clear constants & ranges** | Keep `MIN_… / MAX_…` as `const val` | Guarantees compile‑time safety & easy refactor |
| **Explicit `ValidationResult`** | Use a *sealed* class (`Valid / Invalid`) | Eliminates the “errors list + isValid flag” anti‑pattern and makes pattern‑matching trivial |
| **Default config factory** | `getDefaultConfig()` | One source of truth for defaults |
| **Coercion + fallback** | `createValidatedConfig()` | Users get a *fully valid* config even if they pass bad data |
| **Warnings** | Separate `warnings: List<String>` | Allows callers to log or surface non‑critical issues |
| **Unit‑testable** | All logic in a plain Kotlin class | Easy to write tests for every rule |
