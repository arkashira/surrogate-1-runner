## 1.  What we’re building

A **real‑time smart‑grid telemetry platform** that

| Layer | Responsibility |
|-------|----------------|
| **Ingest** | Pulls streaming sensor data, validates, timestamps, and stores in a short‑term buffer. |
| **Processing** | Applies configurable anomaly detection, enriches alerts, and pushes them to a delivery pipeline. |
| **Delivery** | Routes alerts to the appropriate recipients (SMS, email, SCADA, dashboards) with retry and audit. |
| **Observability** | Exposes metrics, health checks, and a lightweight REST API for status & configuration. |

The goal is a single, well‑tested Python package that can be dropped into an existing grid‑operations stack.

---

## 2.  Core design principles

| Principle | Why it matters | Concrete action |
|-----------|----------------|-----------------|
| **Modularity** | Easy to swap components (e.g., replace the anomaly detector with a ML model). | Expose each layer via a clear interface (`IIngest`, `IProcessor`, `IDelivery`). |
| **Idempotency** | Avoid duplicate alerts when a sensor re‑sends a reading. | Use a deterministic hash of `(timestamp, sensor_id, metric, value)` as the alert key. |
| **Back‑pressure** | Prevent buffer overflow when the downstream system is slow. | Use a bounded `queue.Queue` and a configurable `processing_interval`; drop or log oldest items when full. |
| **Observability** | Operators need to see what’s happening in real time. | Publish Prometheus metrics (`readings_total`, `alerts_total`, `alerts_by_severity`) and a `/health` endpoint. |
| **Extensibility** | Future metrics, thresholds, or delivery channels. | Keep thresholds in a JSON/YAML file that can be reloaded at runtime. |

---

## 3.  Final architecture

+----------------+      +----------------+      +----------------+
|  Sensor Stream | ---> |  Ingest Layer  | ---> |  Processor     |
