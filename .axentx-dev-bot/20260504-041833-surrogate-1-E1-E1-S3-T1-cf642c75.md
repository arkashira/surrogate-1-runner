## Unified Compliance Dashboard – “One‑Stop” Design

Below is a **single, production‑ready implementation** that pulls the strongest ideas from the two candidates:

| Feature | Candidate 1 | Candidate 2 | Unified Choice |
|---------|-------------|-------------|----------------|
| **HTML structure** | Simple table + filter section | Semantic `<section>`s + form | **Semantic, accessible** layout (header, form, dashboard, export) |
| **Filter controls** | `<select>` for account, policy, time‑period | `<select>` + date range | **Hybrid** – keep the `<select>`s for account/policy, add a **date‑range picker** (start/end) for flexible time filtering. |
| **Data fetching** | Inline `fetch` in script | `async/await` helper | **Async/await** for readability + error handling. |
| **Export** | CSV + PDF via `jsPDF` | Only CSV shown | **Both** – CSV (client‑side) and PDF (server‑side for larger data). |
| **Styling** | Basic CSS | None | **Responsive, mobile‑first** CSS with CSS Grid/Flexbox. |
| **Accessibility** | Basic labels | Form semantics | **ARIA** roles, `aria-live` for dynamic updates, `label`‑`for` matching. |
| **Performance** | Re‑fetch on every filter change | No debounce | **Debounce** filter changes to avoid excessive API calls. |
| **Security** | None | None | **CSRF token** included in fetch headers (for Spring‑Boot). |
| **Extensibility** | Hard‑coded options | Dynamic options | **Dynamic option loading** from API. |
