## Unified Real‑Time Cost‑Anomaly Alerting Engine  
*(A single, production‑ready module that blends the strongest ideas from the three candidate proposals.)*

| Feature | Why it matters | How we implement it |
|---------|----------------|---------------------|
| **Ultra‑low latency alerts** | Users need to react within seconds, not minutes. | *Event‑driven architecture* – every incoming cost record is processed immediately via an `EventEmitter`.  Alerts are emitted within **< 1 s** of detection. |
| **Multi‑metric detection** | Cost can spike, budgets can be exceeded, usage patterns can shift, and idle resources can waste money. | We keep **four core rule types**: `percentage_increase`, `budget_limit`, `statistical`, `utilization`.  New rules can be added at runtime. |
| **User‑centric customization** | Different teams have different thresholds, channels, and quiet hours. | A per‑user `Preferences` object stores: <br>• `channels` (email, Slack, webhook, SMS) <br>• `severityFilter` <br>• `quietHours` <br>• `aggregationWindow` <br>• `maxAlertsPerDay` <br>• `language` & `timezone` |
| **Historical context & rolling windows** | Detecting anomalies requires a baseline. | We keep a **rolling 24‑hour window** per user (configurable).  For each rule we use the appropriate window (e.g., last hour for percentage increase). |
| **Graceful degradation** | In high‑frequency scenarios we must avoid memory blow‑up. | `recordCostDataBatch` processes arrays atomically, and the history is pruned to the last 24 h. |
| **Extensible rule engine** | Future metrics (e.g., CPU, network) can be added without touching core logic. | Rules are stored in a `Map`; evaluation is delegated to a `switch` on `rule.type`. |
| **Audit & traceability** | Compliance requires knowing *why* an alert fired. | Every alert payload includes: `ruleId`, `ruleName`, `detectedValue`, `baseline`, `current`, `timestamp`, and `severity`. |
| **Rate‑limiting & quiet hours** | Avoid alert fatigue. | `maxAlertsPerDay` and `quietHours` are enforced before an alert is emitted. |
| **Internationalization** | Teams span multiple time zones. | `timezone` in preferences is used to compute `quietHours` and to format timestamps. |
| **Robust error handling** | Production systems must not crash on malformed data. | All public methods validate input and throw descriptive errors; internal methods catch and log exceptions without propagating. |
