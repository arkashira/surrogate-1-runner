*Run the HTTP server in the main thread and push metric updates from a **daemon thread** so the exporter can be started, stopped and unit‑tested without hanging.*

## 1. Design decisions that were kept / improved  

| What we needed | How the final code satisfies it |
|----------------|--------------------------------|
| **Prometheus‑compatible exporter** | `prometheus_client.start_http_server` is used exactly once, on a configurable port. |
| **Support for many workflows** | `WorkflowMonitor(num_workflows)` builds a dictionary of Gauges **with a `metric_type` label** (`status`, `performance`, …). No need to create a new Gauge object for every metric type. |
| **Easy metric updates** | `update_metric(workflow_id, metric_type, value)` abstracts the label handling. |
| **Periodic simulation / real updates** | A **daemon thread** runs `self._run_loop()` and sleeps `self.update_interval` seconds between cycles. The loop can be replaced by a real data‑collector later. |
| **Non‑blocking start** | `start_monitoring()` returns immediately after launching the background thread – perfect for unit‑tests and for embedding the exporter in larger services. |
| **Graceful shutdown** | `stop_monitoring()` signals the thread to exit and joins it. |
| **Testability** | No infinite `while True` in the main thread, all external calls (`start_http_server`, `time.sleep`) are easy to patch. |
| **Extensibility** | Adding a new metric type is just a matter of calling `update_metric(..., 'new_type', value)`. |
| **Code style & typing** | Full type hints, doc‑strings and PEP‑8 compliance. |
