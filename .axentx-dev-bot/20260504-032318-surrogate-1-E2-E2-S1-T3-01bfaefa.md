## Unified “data‑volume” module  

Below is a single, production‑ready implementation that blends the strongest ideas from the two proposals:

* **YAML‑driven configuration** – a single `sandbox.yaml` file can contain a **global default**, **per‑service overrides**, **high/low traffic profiles**, and **spike parameters**.  
* **Lazy, cached loading** – the file is read once (or re‑loaded on demand) and the result is cached for fast look‑ups.  
* **Typed, dataclass‑backed model** – `VolumeConfig` gives you a clear, IDE‑friendly API while still allowing raw‑dict access for legacy callers.  
* **Convenient helpers** – `get_volume(service_type)` returns the fully‑merged dict, `is_spike_active(service_type)` tells you if spikes are enabled, and `compute_rate(service_type)` gives you the *actual* points‑per‑minute that should be emitted right now (including probabilistic spikes).  
* **Graceful fall‑backs** – missing files, sections, or keys never crash the runner; sensible defaults are applied.

    rate_per_minute: 120

    # Spike configuration – can be overridden per‑service.
    spike:
      enabled: true          # generate spikes?
      factor: 4              # multiply base rate during a spike
      duration_minutes: 3    # how long a spike lasts

      rate_per_minute: 500

  # Per‑service overrides.  Any key present here replaces the corresponding
  # entry from the default block (including the nested ``spike`` dict).
  services:
    web:
      rate_per_minute: 250
      spike:
        enabled: true
        factor: 6
        duration_minutes: 5
        interval_minutes: 20
        chance: 0.10
    db:
      rate_per_minute: 80
      spike:
        enabled: false
    cache:
      rate_per_minute: 150
      spike:
        enabled: true
        factor: 3
        duration_minutes: 2
        interval_minutes: 25
        chance: 0.07
