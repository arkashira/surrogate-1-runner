## 1. Acceptance Criteria
- **Bridge Exposure**: The native module must expose a method `invoke(String modelId, String prompt, ReadableMap options)` that can be called from JavaScript via `NativeModules.Surrogate.invoke`.
- **Rate‑Limit & Cost Tracking**: Each call must be routed through the same rate‑limit and cost‑tracking logic as the server SDK. The module should forward the request to the existing `SurrogateClient` and return the same response structure.
- **Error Propagation**: Any error thrown by the backend (e.g., network failure, rate‑limit exceeded, invalid model) must be caught and re‑thrown as a JavaScript exception (`Promise.reject`) that the workflow engine can catch.
- **Promise Based API**: The bridge method must return a `Promise` to JavaScript, resolving with the LLM response or rejecting with an error.
- **Options Handling**: The `options` map may contain optional fields such as `temperature`, `maxTokens`, etc. These must be forwarded to the SDK call without modification.
- **Threading**: The bridge must execute network calls off the UI thread and resolve/reject on the JavaScript thread.
- **Logging**: Each invocation should log the `modelId` and a truncated `prompt` (first 100 chars) for debugging.

    expect(response).toEqual(mockResponse);
  });

  test('rejects with backend error', async () => {
    const error = new Error('Rate limit exceeded');
    Surrogate.invoke.mockRejectedValue(error);

  });
