| Decision | Chosen approach | Why |
|--|--|--|
| **SLA definition** | `slaMs = latestStatusTimestamp - createdAt` (never negative) | Matches PRD and is deterministic. If no status exists, use `now` so SLA keeps increasing. |
| **Timeline ordering** | Store entries in insertion order; sort by `timestamp asc` on read; enforce monotonic non-decreasing timestamps at write | Guarantees stable ordering and avoids timeline corruption from clock skew. |
| **Status transition source of truth** | Single writer in `slaService` that validates and records; updates request current status atomically | Prevents divergence between timeline and current status. |
| **Actor identity** | Prefer `X-Actor` header; fallback to `system`; reject empty/whitespace | Clear auditability and safe default. |
| **Persistence scope** | In‑memory maps for demo; interfaces designed so stores can be swapped for DB later | Fast to implement and test, but not a dead end. |
| **Idempotency** | Client-supplied `eventId` optional; when provided, deduplicate by `(requestId, eventId)` | Prevents duplicate transitions from retries (critical for production). |
| **Timestamp handling** | Accept optional `timestamp` (ISO8601); validate; reject future timestamps beyond small leeway (e.g., 30s) | Prevents accidental backfill errors and clock skew issues. |
| **Error model** | 400 for bad input, 404 for missing request, 409 for invalid transition or duplicate eventId | Standard HTTP semantics and actionable for clients. |
