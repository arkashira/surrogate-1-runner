## 1. Design overview  

| Concern | How it is handled | Why this choice |
|---------|------------------|-----------------|
| **Telemetry persistence** | Thread‑safe JSON‑line file *and* an in‑memory aggregator that periodically emits a summary (e.g. to stdout, a monitoring system, or a remote collector). | Gives both raw audit logs (candidate 1) **and** quick operational insight (candidate 2). |
| **Throttling decision** | Simple time‑based back‑off that respects a *critical* flag, the current monitor refresh rate, and two environment‑tuned knobs (`AXENTX_THROTTLE_REFRESH_HZ`, `AXENTX_THROTTLE_FACTOR`). | Keeps the deterministic logic from candidate 1, but adds a tiny cache to avoid repeated `time.time()` calls for the same task. |
| **Public API** | `run_inference(task_name, inference_fn, critical=False)` – a thin wrapper that does everything: decide, log, and return the result (or `None` when throttled). | Mirrors candidate 1’s `run_inference` while exposing the logger as a singleton (candidate 2). |
| **Configuration** | All knobs are read from environment variables at import time, with sensible defaults and a single place (`config.py`). | Centralises settings, making them easy to document and test. |
| **Thread‑safety** | A global `threading.Lock` protects file writes; a per‑process singleton protects the in‑memory collector. | Guarantees correct behaviour in multi‑threaded inference servers. |
| **Graceful shutdown** | `PerformanceLogger.shutdown()` flushes the final summary and stops the background reporter thread. | Prevents loss of the last interval’s data when the process exits. |
