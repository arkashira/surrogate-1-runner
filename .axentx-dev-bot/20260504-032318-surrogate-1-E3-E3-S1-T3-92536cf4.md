## Unified, Production‑Ready Logging & Sandbox Runtime  

Below is a **single, self‑contained implementation** that merges the strongest ideas from the two candidate proposals:

| What we keep | Why it’s the best |
|--------------|-------------------|
| **`StructuredLogger`** – a tiny, thread‑safe logger that writes **JSON‑lines** directly to a stream. | Guarantees deterministic, single‑line output without the overhead of the full `logging` package and works even when the process is started with a very low‑level `PYTHONPATH`. |
| **`get_structured_logger()`** wrapper that returns a **singleton** `logging.Logger` configured with the same JSON formatter. | Lets existing code that already uses the standard `logging` API continue to work unchanged. |
| **`log_event()`** helper that normalises level strings and injects arbitrary `extra` fields. | Provides a convenient, error‑proof way to emit structured logs from any part of the code base. |
| **`SandboxError`** – a domain‑specific exception used to differentiate user‑errors from unexpected crashes. | Makes CI/CD exit‑code handling explicit and easy to test. |
| **`run_sandbox()`** – the high‑level entry point that logs start / success / failure events, returns **four distinct exit codes** (0, 1, 2, 3) and is safe to call from a CI/CD step. | Gives pipelines deterministic behaviour and a clear contract for downstream tooling. |
| **`_run_docker_image()`** – a reusable helper that launches a Docker container, streams its stdout/stderr as structured logs, and returns the container’s exit code. | Provides the concrete “sandbox” execution model that the original second proposal wanted, while still using the unified logger. |
| **Environment‑driven log level** (`AXENTX_LOG_LEVEL`) and **timestamp with millisecond precision**. | Makes the logger configurable without code changes and satisfies most observability platforms. |
