## 1️⃣  One‑Page Scheduler Specification

| Item | Description | Why it matters |
|------|-------------|----------------|
| **1. Storage** | Cron schedules live in a relational table `workflow_schedules` (PostgreSQL/SQLite). Columns: `id (PK)`, `workflow_id (text)`, `workflow_version (int)`, `cron_expression (text)`, `enabled (bool)`, `created_at`, `updated_at`. | Guarantees durability, auditability, and easy querying. |
| **2. Workflow Identity** | A workflow is identified by a *compound key* `<workflow_id> + <workflow_version>`. The scheduler stores the full key so that a new version can be scheduled independently of older ones. | Prevents accidental re‑execution of the wrong version. |
| **3. Runner Invocation** | The scheduler calls the **workflow runner** via a **REST endpoint** (`POST /workflows/run`). Payload: `{ "workflow_id": "...", "workflow_version": ... }`. The endpoint is protected by a short‑lived API key. | Keeps the scheduler stateless, allows horizontal scaling, and decouples scheduling from execution. |
| **4. Cron Validation** | On add/update, the cron string is parsed with `CronTrigger.from_crontab`. If parsing fails, the operation is rejected and an error is logged. | Avoids silent failures and mis‑scheduled jobs. |
| **5. Error Handling** | 1. **Mis‑configured cron** – reject with HTTP 400 and log. 2. **Runner timeout** – retry up to 3 times with exponential back‑off. 3. **Database failure** – log and raise a custom `SchedulerDBError`. | Makes the system resilient and observable. |
| **6. Test Expectations** | • Unit tests for CRUD operations on schedules. <br>• Integration test that starts the scheduler, adds a schedule, and verifies the runner endpoint is hit at the expected time (using a mocked clock). <br>• Load test that adds 10 000 schedules and checks that the scheduler remains responsive. | Ensures correctness, performance, and reliability. |
