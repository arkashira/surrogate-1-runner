  returns partial results with a TimeoutError for the unfinished
  agents.
* Simple Agent interface: an agent must expose an async `run(request)`
  coroutine that returns a string result.

-----

## 1. Acceptance Criteria  

1. **Deterministic Ordering**  
   * Given a list of agents and a request ID, the orchestrator returns the agents sorted in a repeatable order that depends only on the request ID and agent identifiers.  
   * The order must be identical for the same request ID across multiple runs.  

2. **Concurrent Execution & Latency**  
   * When orchestrating 3, 5, or 7 agents, the total wall‑time from start to completion must be **≤ 200 ms** under simulated load (agents sleep ≤ 50 ms each).  
   * The orchestrator must launch all agents in parallel (not sequentially).  

3. **Timeout Handling**  
   * If any agent exceeds a configurable timeout (default 150 ms), the orchestrator must cancel that agent and continue with the others.  
   * The orchestrator must return a result indicating which agents timed out.  

4. **DummyAgent Functionality**  
   * `DummyAgent` must expose a `run()` coroutine that returns a deterministic string after a configurable delay.  
   * The agent must be usable by the orchestrator without external dependencies.  

5. **Extensibility**  
   * The orchestrator must accept any object implementing a `run()` coroutine signature.  
   * Adding a new agent type should not require changes to the orchestrator logic.  

6. **Documentation & Usage Example**  
   * The module must contain a docstring explaining the deterministic algorithm and a short example that can be executed from the command line.
