# surrogate‑1‑runner  

Parallel public‑dataset ingest workers for the  
[axentx/surrogate‑1‑training‑pairs](https://huggingface.co/datasets/axentx/surrogate-1-training-pairs)  
HuggingFace dataset.

## What this does  

* Every **30 minutes** (or on a manual `workflow_dispatch`) GitHub Actions launches **16 parallel runners**.  
* Each runner processes a **deterministic 1/16 slice** of the public‑dataset list (`slug‑hash bucket = SHARD_ID`).  
* The slice is defined in `bin/dataset-enrich.sh`.  
* For each entry the runner:  
  1. **Streams** the raw record.  
  2. **Normalises** it to the repository schema.  
  3. **Deduplicates** via the central MD5‑hash store.  
  4. **Uploads** the cleaned record to a unique path inside the dataset repo.  

The result is a continuously‑growing, de‑duplicated, schema‑validated training‑pair collection.

---

## SDK Usage  

The **`axentx‑surrogate` SDK** ships a tiny, synchronous helper `run_workflow` that hides the HTTP request needed to trigger the ingestion orchestration.

```python
# synchronous usage – simplest possible
from axentx_surrogate import SurrogateClient

client = SurrogateClient(base_url="https://api.axentx.com")
definition = {
    "workflow": "ingest",
    "parameters": {"shard_id": 3}   # 0‑15 inclusive
}
correlation_id = client.run_workflow(definition)
print(f"Workflow started, correlation ID: {correlation_id}")
```

> **Why a client class?**  
> It lets you configure authentication headers, time‑outs, and retry policies in one place, while still exposing a single `run_workflow` method.

---

## Async usage example  

Many modern code‑bases (FastAPI, Quart, background workers) run under **`asyncio`**.  
The SDK itself stays **synchronous** (so it can be used from plain scripts), but you can call it from an async context without blocking the event loop by wrapping it in an executor.

```python
import asyncio
from concurrent.futures import ThreadPoolExecutor
from axentx_surrogate import SurrogateClient

# ----------------------------------------------------------------------
# Helper that runs the *blocking* SDK call in a thread‑pool
# ----------------------------------------------------------------------
def _trigger_sync(client: SurrogateClient, definition: dict) -> str:
    """Blocking call – kept separate for clarity."""
    return client.run_workflow(definition)

# ----------------------------------------------------------------------
# Public async wrapper
# ----------------------------------------------------------------------
async def trigger_workflow_async(
    client: SurrogateClient,
    definition: dict,
    *,
    max_workers: int = 1,
) -> str:
    """
    Run ``client.run_workflow`` without blocking the event loop.

    Parameters
    ----------
    client:
        An instantiated :class:`SurrogateClient`.
    definition:
        The workflow definition dict (see the synchronous example).
    max_workers:
        Number of threads in the temporary pool; default 1 because the
        call is I/O‑bound and short‑lived.

    Returns
    -------
    str
        The correlation ID returned by the API.
    """
    loop = asyncio.get_running_loop()
    # A fresh ThreadPoolExecutor is cheap for a single short task.
    with ThreadPoolExecutor(max_workers=max_workers) as pool:
        correlation_id = await loop.run_in_executor(
            pool,
            _trigger_sync,
            client,
            definition,
        )
    return correlation_id
