# Surrogate‑1 Workflow Templates Index

> A single, searchable entry point for every automated workflow in the Surrogate‑1 project.  
> All workflows are **idempotent**, **deterministic**, and follow the same conventions so you can add or modify them with confidence.

---

## 📦 Available Workflows

| Workflow | Description | Trigger | Last Updated |
|----------|-------------|---------|--------------|
| [dataset‑ingest.yml](./dataset‑ingest.yml) | Parallel ingestion of public HuggingFace data (16 runners). | `schedule (*/30 * * * *)` / `workflow_dispatch` | 2026‑05‑04 |
| [data‑validation.yml](./data‑validation.yml) | Schema & duplicate validation of ingested shards. | `workflow_dispatch` | 2026‑05‑04 |
| [shard‑rebalance.yml](./shard‑rebalance.yml) | Redistribute data when runners are added/removed. | `workflow_dispatch` | 2026‑05‑04 |
| [cache‑cleanup.yml](./cache‑cleanup.yml) | Remove stale cache entries & orphaned artifacts. | `schedule (daily)` / `workflow_dispatch` | 2026‑05‑04 |
| [dataset‑publish.yml](./dataset‑publish.yml) | Publish validated data to the HuggingFace dataset repo. | `workflow_run` (after successful ingest & validation) | 2026‑05‑04 |

> **Tip:** Use the *search bar* in this file to jump to a workflow quickly (`Ctrl+F` → workflow name).

---

## 🚀 Quick Start

### 1. Trigger a Full Ingest (all shards)

```bash
gh workflow run dataset-ingest.yml
```

### 2. Trigger a Single Shard

```bash
gh workflow run dataset-ingest.yml -f shard_id=7
```

### 3. Run Validation Manually

```bash
gh workflow run data-validation.yml -f validate_schema=true -f check_duplicates=true
```

### 4. Rebalance Shards After Adding a Runner

```bash
gh workflow run shard-rebalance.yml -f add_runner=true -f target_shard=3
```

---

## 📌 Workflow Conventions (All Workflows)

| # | Convention | Why it matters |
|---|------------|----------------|
| 1 | **Idempotency** | Safe to re‑run; no side‑effects on repeated execution. |
| 2 | **Deterministic Sharding** | `slug‑hash % 16 = SHARD_ID` ensures even distribution. |
| 3 | **Fail‑Fast Validation** | Schema/duplicate checks run before any upload. |
| 4 | **Atomic Commits** | Each shard uploads to a unique path (`shard‑{ID}`) to avoid collisions. |
| 5 | **Centralized Secrets** | All workflows read `HF_TOKEN` from the same secret store. |
| 6 | **Explicit Inputs** | Every workflow exposes `workflow_dispatch` inputs for manual overrides. |

## 🔧 Troubleshooting

| Issue | Quick Fix |
|-------|-----------|
| Runner failures | Check the Actions tab → Logs → Verify `HF_TOKEN` and network connectivity. |
| Duplicate data | Run `data-validation.yml` with `check_duplicates=true`. |
| Schema errors | Inspect `bin/dataset-enrich.sh` for recent schema changes. |
| Shard imbalance | Execute `shard-rebalance.yml` with `add_runner`/`remove_runner`. |
| Stale cache | Run `cache-cleanup.yml` with `full_cleanup=true`. |
