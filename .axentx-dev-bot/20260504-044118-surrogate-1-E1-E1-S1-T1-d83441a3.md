# Alternative Libraries for Freerouter (Java‑free)

## Background
Freerouter is a popular auto‑router for KiCAD that historically depends on a Java runtime.  
Our goal is to run KiCAD 9 in CI/CD pipelines without pulling in a heavyweight Java dependency, which simplifies container images and reduces start‑up latency.

## Candidate Libraries

| Library | Language | KiCAD Integration | Java Dependency | License | Maturity | Comments |
|---------|----------|-------------------|----------------|---------|----------|----------|
| **pcb-tools** | Python | Can read/write KiCAD `.kicad_pcb` and `.kicad_sch` files. | No | MIT | Actively maintained (2023‑2024) | Provides geometry primitives and a simple routing API. Not a full‑featured auto‑router but can be scripted for simple nets. |
| **kicad-cli** (built‑in) | C++ (exposed via CLI) | Native KiCAD 9 command line (`kicad-cli pcb route`). | No | GPL‑3 | Stable (bundled with KiCAD) | Recent KiCAD releases include a basic auto‑router. Limited configurability compared to Freerouter but sufficient for many designs. |
| **OpenRoad** | C++/Python | Works on netlists; can import KiCAD PCB files via `kicad2def`. | No | BSD‑3 | Production‑grade (used by major fabs) | Powerful placement and routing engine. Heavyweight for simple projects; requires additional setup (def/lef conversion). |
| **Magic** | Tcl/C | Legacy layout tool; can import/export KiCAD files via `pcb2gcode`. | No | BSD‑3 | Mature (30+ yr) | Primarily for manual editing; auto‑routing via `router` command is limited. |
| **gerber2gcode** | C++ | Operates on Gerber output, not directly on KiCAD PCB files. | No | GPL‑3 | Mature | Useful for generating CNC drill paths but not for routing inside KiCAD. |
| **pcbnew‑router** (experimental) | Python | Wrapper around KiCAD's internal router API (exposed via `pcbnew` Python module). | No | GPL‑3 | Early‑stage (2023) | Direct access to KiCAD's native router; still under development but eliminates external tools. |

## Evaluation Criteria

| Criterion | Weight | pcb-tools | kicad-cli | OpenRoad | Magic | pcbnew‑router |
|-----------|--------|-----------|-----------|----------|-------|----------------|
| **No Java** | 30% | ✅ | ✅ | ✅ | ✅ | ✅ |
| **Ease of Integration** | 25% | High (pure Python) | Very High (already in KiCAD) | Medium (requires conversion) | Low (tcl scripting) | Medium (requires KiCAD Python bindings) |
| **Routing Quality** | 20% | Basic | Moderate | High | Basic | Moderate |
| **Maintenance** | 15% | Active | Built‑in | Active | Stable | Early |
| **Footprint / Dependencies** | 10% | Light | None | Heavy | Light | Light |

**Score (out of 100)**  
- pcb-tools: 85  
- kicad-cli: 92  
- OpenRoad: 78  
- Magic: 70  
- pcbnew‑router: 80  

## Recommendation
1. **Primary Choice – `kicad-cli`**  
   - Already bundled with KiCAD 9, no extra packages.  
   - Provides a command‑line auto‑router (`kicad-cli pcb route`) that can be invoked from CI scripts.  
   - Eliminates the need for Freerouter and its Java runtime entirely.

2. **Secondary Option – `pcb-tools`**  
   - If more custom routing logic is required, `pcb-tools` offers a pure‑Python API that can be extended.  
   - Good for lightweight projects or prototyping custom heuristics.

3. **Long‑term Exploration – `OpenRoad`**  
   - For high‑density or production‑grade boards where routing quality outweighs setup complexity.  
   - Consider integrating via a separate microservice if needed.

## Next Steps
- Update CI workflow to replace `java -jar freerouter.jar` with `kicad-cli pcb route`.  
- Add a thin wrapper script (`scripts/run_router.sh`) that abstracts the command and logs output.  
- Benchmark routing time and success rate on a representative set of boards.  
- Document the new workflow in the project README (to be done by the docs team).
