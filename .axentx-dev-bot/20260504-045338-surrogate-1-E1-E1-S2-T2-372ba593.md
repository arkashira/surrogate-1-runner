## 1. Acceptance Criteria
1. **Azure Authentication** – The `AzureResourceMonitor` must successfully authenticate to an Azure tenant using a service principal and obtain a valid access token (HTTP 200, token non‑empty) within 5 seconds.  
2. **Resource Discovery** – After authentication, the monitor must retrieve a complete list of resources for at least three Azure resource types (e.g., Virtual Machines, Storage Accounts, Key Vaults) and return a non‑empty collection when resources exist.  
3. **Drift Detection** – When a resource’s configuration deviates from the stored baseline, the monitor must emit a `DriftEvent` containing the resource ID, the property that drifted, and the before/after values.  
4. **Compliance Violation Detection** – For any resource that violates a predefined compliance rule (e.g., “VM must have encryption enabled”), the monitor must generate a `ComplianceViolation` with severity ≥ `MEDIUM`.  
5. **Real‑Time Streaming** – Detected drift and compliance events must be published to the internal event bus within 2 seconds of detection, preserving ordering per resource.  
6. **Error Handling** – Invalid credentials, network timeouts, or Azure API rate‑limit responses must be captured, logged, and cause the monitor to retry up to 3 times with exponential back‑off before marking the monitor as `UNHEALTHY`.  
7. **Performance** – Under a load of 500 concurrent Azure resources, the end‑to‑end latency from change detection to event publication must not exceed 5 seconds on average.
