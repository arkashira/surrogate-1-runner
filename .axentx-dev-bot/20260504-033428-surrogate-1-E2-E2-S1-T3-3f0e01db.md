## 1.  What we’re building

A **web‑based cost‑forecast dashboard** that lets a user pick a time horizon (7 days, 30 days, 90 days, 180 days) and see a table of projected costs for each day in that horizon.  
The page is a single‑page HTML/JS file that talks to a backend API (`/api/forecast`) and shows a loading spinner while the data is being fetched.

---

## 2.  Safety & Responsible‑AI considerations

| Issue | Why it matters | How we address it |
|-------|----------------|-------------------|
| **Data privacy** | Forecast data may contain sensitive financial information. | The API should only return the minimal JSON needed (`date`, `cost`). No user credentials or PII are sent. |
| **Rate‑limit abuse** | A malicious user could hammer the API. | The client only makes one request per period change; the server should enforce a per‑IP or per‑API‑key rate limit. |
| **Error handling** | Users should not see raw stack traces. | The UI displays a friendly “Error loading forecast data.” message and logs the technical error to the console for developers. |
| **Content‑security** | Prevent injection of malicious scripts. | The page uses `textContent` for all dynamic data; no `innerHTML` is used. The server should set `Content‑Security‑Policy` headers. |
| **Accessibility** | All users can interact with the UI. | The `<label>` is properly associated with the `<select>`. The loading indicator uses `display: none`/`block` so screen readers can detect changes. |
