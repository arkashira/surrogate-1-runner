## 1. Acceptance Criteria  
- **Automatic Detection** – The system must expose an endpoint `/api/config` that returns a JSON array containing every default configuration key, its default value, and the current (possibly overridden) value.  
- **Override Capability** – A POST to `/api/config` with a JSON payload `{ "key": "<configKey>", "value": "<newValue>" }` must update the in‑memory override map and return the updated configuration entry.  
- **UI Rendering** – The `config.html` page must render a table row for each configuration entry with columns: *Key*, *Default Value*, *Current Value*, *Override Input*.  
- **UI Interaction** – Changing the value in the override input and submitting the form must trigger a POST to `/api/config` and refresh the table to reflect the new value.  
- **Transparency** – The UI must clearly indicate whether a value is the default or has been overridden (e.g., a badge or different styling).  
- **Validation** – The backend must reject invalid keys or values (e.g., empty string) with a 400 Bad Request and an explanatory message.  
- **Documentation** – The code comments and API documentation (e.g., Swagger annotations) must describe the override mechanism and the default configuration map.
