## Final, “best‑of‑both‑worlds” Google‑Ads‑Connection UI

Below is a single, production‑ready implementation that pulls the strongest ideas from the two candidates:

* **Clean separation of concerns** – HTML, CSS, and JavaScript live in their own files.  
* **Real OAuth flow** – the UI redirects to Google’s OAuth endpoint and the server exchanges the code for a refresh token.  
* **Secure token revocation** – the disconnect button calls a protected API that revokes the token on the server.  
* **Responsive design** – the CSS is mobile‑first and uses flexbox for centering.  
* **State persistence** – the page reads the current connection status from the server on load, so the UI is always in sync.  
* **Accessibility** – semantic elements, `aria-live` for status updates, and focus management.  

> **Assumptions**  
> * The backend exposes the following endpoints:  
>   * `GET /api/google-ads/status` → `{ connected: boolean, email?: string }`  
>   * `POST /api/google-ads/connect` → `{ success: boolean, email?: string }` (expects `{ code }` in body)  
>   * `POST /api/google-ads/disconnect` → `{ success: boolean }`  
> * The OAuth client ID, redirect URI, and scopes are configured on the server; they are referenced in the JavaScript via environment‑specific constants.

            </div>
        </section>
    </main>

    <script src="google_ads_connection.js" defer></script>
</body>
</html>
```

*Why this is better?*  
* Uses `hidden` instead of `display:none` so screen readers don’t see the element.  
* `aria-live="polite"` ensures status changes are announced.  
* The `<main>` element gives a clear semantic root.
