## 1. Acceptance Criteria
1. **Historical baseline calculation** – The system must compute a statistical baseline (e.g., moving average and standard deviation) from at least the past 30 days of cost data for each cloud resource group.  
2. **Anomaly threshold** – An anomaly is flagged when the current day's cost exceeds the baseline by **≥ 3 σ** (or a configurable percentage) for that resource group.  
3. **Alert delivery** – When an anomaly is detected, an alert is sent **within 5 minutes** to the configured channel(s): email (SMTP) **and/or** Slack (Webhook).  
4. **Dashboard integration** – Detected anomalies appear on the real‑time cost monitoring dashboard with timestamp, resource group, actual cost, baseline, and deviation percentage.  
5. **Configurable parameters** – The detection window (days), sigma threshold, and alert destinations are configurable via the platform’s `config.yaml` without code changes.  
6. **Idempotent processing** – Re‑running the detection for a given day must not generate duplicate alerts.  
7. **Observability** – Each detection run logs a structured entry (`detection_id`, `resource_group`, `status`, `alert_sent`) to the platform’s logging service.
