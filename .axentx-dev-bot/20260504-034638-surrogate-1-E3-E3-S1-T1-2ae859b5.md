## 1. Acceptance Criteria
- **Integration Hook**: The system exposes a public method `registerWithPipeline(String pipelineId)` that can be invoked by the CI/CD orchestrator without errors.
- **Triggering Assessment**: When a build starts, the integrator automatically invokes `runVulnerabilityAssessment()` exactly once per build.
- **Result Retrieval**: The assessment results are returned as a JSON object containing at least the fields `buildId`, `timestamp`, `vulnerabilities` (array), and `status` (`PASS`/`FAIL`).
- **Dashboard Update**: The integrator publishes the results to the CI/CD dashboard via a REST endpoint `POST /ci/dashboard/results` and receives an HTTP 200 OK response.
- **Error Handling**: If the vulnerability scanner fails, the integrator logs the error, returns a `status: FAIL`, and still posts to the dashboard with an empty `vulnerabilities` array.
- **Performance**: The entire assessment and posting process completes within 30 seconds for a build with up to 2000 lines of code.
- **Security**: All data transmitted to the dashboard is signed with an HMAC using a shared secret and verified by the dashboard service.
