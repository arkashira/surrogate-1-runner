# Surrogate‑1 Drift Detection – Unified Monitoring & Alerting Guide

> **Audience** – DevOps, SRE, and ML Ops teams that run Surrogate‑1 in production.  
> **Goal** – Expose Prometheus metrics, visualise them in Grafana, and fire actionable alerts that reach the right people (Slack, PagerDuty, email, etc.).

> **Assumptions** –  
> * Surrogate‑1 is already running and exposes a `/metrics` endpoint on the runner process (default port `8000`).  
> * You have a Prometheus server, Grafana instance, and an Alertmanager deployment available.  
> * You have the necessary credentials for Slack, PagerDuty, or email.

---

## 1. Prometheus Integration

### 1.1 Exported Metrics (most complete set)

| Metric | Type | Description |
|--------|------|-------------|
| `surrogate_drift_total` | Counter | Total number of drift events detected. |
| `surrogate_drift_current{shard_id}` | Gauge | Current drift status for a shard (`1` = drift, `0` = healthy). |
| `surrogate_drift_processing_seconds` | Histogram | Processing latency per shard. |
| `surrogate_errors_total` | Counter | Number of errors while evaluating drift. |
| `surrogate_drift_events_total` | Counter | Alias for `surrogate_drift_total` (kept for backward‑compatibility). |

> **Tip** – Keep both `surrogate_drift_total` and `surrogate_drift_events_total` if you’re upgrading from an older Surrogate‑1 release.

### 1.2 Scrape Configuration

Add a single job to `prometheus.yml`:

```yaml
scrape_configs:
  - job_name: 'surrogate_drift'
    static_configs:
      - targets: ['<SURROGATE_HOST>:8000']   # replace with your runner host(s)
    metrics_path: /metrics
    scheme: http
```

If you run multiple runners, list each host/port in `static_configs`.

> **Why 8000?** – Surrogate‑1’s default HTTP server port. Change if you re‑configured the exporter.

---

## 2. Grafana Dashboard

A ready‑to‑import JSON dashboard is shipped with the repo:

* `grafana/surrogate_drift_dashboard.json` (or `surrogate-1.json` – both are identical).

### Import Steps

1. Open Grafana → **+** → **Import**.  
2. Upload the JSON file or paste its contents.  
3. Choose the Prometheus data source you configured above.  
4. Click **Import**.

The dashboard contains panels for:

| Panel | What it shows |
|-------|---------------|
| `Total Drift Events` | Line chart of `surrogate_drift_total`. |
| `Per‑Shard Drift Heatmap` | Heatmap of `surrogate_drift_current`. |
| `Processing Latency` | Histogram of `surrogate_drift_processing_seconds`. |
| `Error Rate` | Counter of `surrogate_errors_total`. |

> **Customisation** – If you need a different time range or additional panels, duplicate the JSON and tweak the queries.

---

## 3. Alerting Rules

### 3.1 Prometheus Alert Rules

Create `prometheus/alerts/surrogate_drift.rules.yml`:

```yaml
groups:
- name: surrogate_drift
  rules:
  - alert: SurrogateShardDrift
    expr: surrogate_drift_current == 1
    for: 5m
    labels:
      severity: critical
    annotations:
      summary: "Drift detected on shard {{ $labels.shard_id }}"
      description: "Surrogate‑1 has detected data drift on shard {{ $labels.shard_id }} for >5 minutes."

  - alert: SurrogateHighLatency
    expr: histogram_quantile(0.95, sum(rate(surrogate_drift_processing_seconds_bucket[5m])) by (le)) > 30
    for: 2m
    labels:
      severity: warning
    annotations:
      summary: "High processing latency"
      description: "95th percentile processing latency > 30s over the last 5 minutes."

  - alert: SurrogateProcessingError
    expr: surrogate_errors_total > 0
    for: 1m
    labels:
      severity: critical
    annotations:
      summary: "Surrogate‑1 processing error"
      description: "Errors have been recorded for the last minute."
```

> **Why these rules?**  
> * `SurrogateShardDrift` – immediate, shard‑level drift.  
> * `SurrogateHighLatency` – performance degradation.  
> * `SurrogateProcessingError` – operational failures.
