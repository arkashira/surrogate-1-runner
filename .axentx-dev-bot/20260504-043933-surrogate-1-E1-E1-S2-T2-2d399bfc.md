## 1. Acceptance Criteria
- **AC‑1**: A client can open a WebSocket connection only when presenting a valid JWT in the `Sec-WebSocket-Protocol` header (or a custom `Authorization` header). Connection is rejected (HTTP 101 not sent) for missing/invalid tokens.  
- **AC‑2**: The JWT must be signed with the server’s RSA/ECDSA private key and contain the claim `sub` equal to the user’s identifier and `exp` not in the past.  
- **AC‑3**: Upon successful authentication, the server assigns the client to a private channel named `session:{sessionToken}` where `sessionToken` is a UUID stored in the JWT claim `sid`.  
- **AC‑4**: Execution‑completion events are published to the client’s private channel as a JSON payload:
  ```json
  {
    "txHash": "<0x…>",
    "blockNumber": 123456,
    "gasUsed": 21000
  }
  ```
- **AC‑5**: The payload is delivered within 200 ms of the internal “execution completed” event (measured from server publish to client receipt).  
- **AC‑6**: The server logs authentication failures with the request IP and reason, without leaking token contents.  
- **AC‑7**: The JWT verification logic is covered by unit tests achieving ≥ 90 % branch coverage.
