| What we keep | Why it is valuable |
|--------------|--------------------|
| **CLI‑first design** (`FeedbackForm` that can be called from an IDE plug‑in) | Works in any environment, no GUI dependencies, matches the original “IDE integration point”. |
| **Optional Tkinter GUI** (`show_feedback_form`) | Gives developers a friendly window when a graphical environment is available. |
| **Centralised JSON store** (`data/feedback.json`) **plus per‑file “outbox”** (`~/.axentx/feedback/feedback_*.json`) | The central file is perfect for quick local analysis; the outbox lets a background worker ship feedback to the AxentX SaaS endpoint without blocking the UI/CLI. |
| **Robust file handling** (`_ensure_storage_file`, atomic writes, error handling) | Guarantees the storage directory exists and protects against corrupted JSON. |
| **Background handler** (`FeedbackHandler`) that retries on failure | Guarantees eventual delivery even when the network is flaky. |
| **Clear, typed API** (`collect_feedback`) | Easy to call from any IDE plug‑in (Python, VS Code, IntelliJ, etc.). |
| **Extensible data model** (rating + type + comments + timestamp) | Supports both the numeric rating from the CLI and the richer “feedback type” dropdown from the GUI. |
| **No hard‑coded paths** – everything is relative to the package or the user’s home directory. | Makes the code portable across dev machines and CI runners. |
