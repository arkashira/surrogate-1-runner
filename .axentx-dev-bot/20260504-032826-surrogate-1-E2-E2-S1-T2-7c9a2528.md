## 1. Acceptance Criteria  

- **API Trigger** – A `POST /workflow/{id}/pause` endpoint must return HTTP 200 and set the workflow state to `PAUSED`.  
- **Command Trigger** – Executing `surrogate pause {id}` via CLI must set the workflow state to `PAUSED` and return success.  
- **State Persistence** – After pausing, the workflow’s current step, variables, and any pending actions must be stored in the persistence layer (e.g., database or file).  
- **Resume API** – A `POST /workflow/{id}/resume` endpoint must return HTTP 200 and change the state from `PAUSED` to `RUNNING`.  
- **Resume Command** – Executing `surrogate resume {id}` must restore the workflow to `RUNNING` and continue execution from the paused step.  
- **Idempotency** – Re‑pausing an already paused workflow or re‑resuming a running workflow must return a 409 Conflict with an explanatory message.  
- **Error Handling** – Pausing a non‑existent workflow returns 404; attempting to pause a completed workflow returns 400.
