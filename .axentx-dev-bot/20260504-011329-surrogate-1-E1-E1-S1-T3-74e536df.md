The goal is to give developers an **instant, human‑readable description** (and a short remediation hint) for every vulnerability identifier that the scanner emits, while also offering a **service‑oriented API** that can be extended at run‑time and is fully unit‑tested.

### 1. Design goals taken from the two proposals  

| Goal | Source | How it is realised |
|------|--------|--------------------|
| **Simple, fast lookup** | Candidate 1 | A module‑level dictionary (`_EXPLANATIONS`) that is accessed in O(1). |
| **Case‑insensitive, graceful fallback** | Candidate 1 | Normalise the key (`.strip().upper()`) and return a generic message when the id is unknown. |
| **Extensibility / add‑on at runtime** | Candidate 2 | A thin service class (`VulnerabilityExplanationService`) that wraps the same dictionary and exposes `add_explanation`. |
| **Rich object model (id + text)** | Candidate 2 | A tiny `VulnerabilityExplanation` data class so callers can inspect the id and the text separately. |
| **Comprehensive documentation & type hints** | Both | Docstrings, type‑annotated signatures and inline comments. |
| **Automated verification** | Candidate 2 | A small pytest‑compatible test‑suite that covers lookup, fallback and dynamic addition. |
| **Single source of truth** | Both | All explanations live in one dict; the service simply reads/writes that dict, guaranteeing that `get_explanation` (functional API) and the service stay in sync. |
