### 2.1  `core.py`

* `TestCaseResult` – unchanged from Candidate 1.
* `TestSuiteSummary` – now **only** aggregates *case‑level* data; no suite‑level counters.
* `parse_junit_xml`, `parse_pytest_xml`, `parse_all` – each returns a `TestSuiteSummary`.  
  *The parsing logic is identical to Candidate 1 but the counting is fixed.*  
* `parse_all` accepts a list of file paths and a format hint (`"junit"` or `"pytest"`).  
  If the format is omitted, the file extension is used (`.xml` → JUnit, `.pytest.xml` → PyTest).

### 2.2  `github.py`

* `_github_annotation` builds a **valid** GitHub Actions annotation string:

```python
f"::error file={case.file},line={case.line},title={title}::{msg}"
```

* `emit_annotations(summary)` – emits all annotations in the order they appear in the log.

### 2.3  `cli.py`

* Uses `argparse` to accept:
  * `--format {junit,pytest}` – optional.
  * `--json` – optional flag to write a JSON summary to `stdout` (or a file).
  * One or more XML files.
* Returns an **exit code** (`0` if all tests passed, `1` if any failure/error, `2` on parse error) – *no `sys.exit` inside the library*.
* The `main()` function is the entry point for `python -m ci_reporting`.

### 2.4  `json_export.py`

* `to_json(summary)` – serialises a `TestSuiteSummary` to a compact JSON string.
* Useful for downstream GitHub Actions steps that need structured data.

### 2.5  Tests

* `test_core.py` – verifies correct parsing and counting for both JUnit and PyTest XML.
* `test_github.py` – checks that annotations are correctly formatted and that only failures/errors produce annotations.
* `test_cli.py` – runs the CLI with temporary XML files and asserts exit codes and output.
