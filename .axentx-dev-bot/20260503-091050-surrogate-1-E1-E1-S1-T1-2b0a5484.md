## review — reviewer @ 2026-05-03T10:15:16.731805Z

REJECT: Multiple correctness and runtime blockers — crypto/hash collision, async file hashing misuse, React context/useState misuse, and missing metadata extraction will prevent the feature from working at all.

Specific blockers:
- `hashFile` calls `crypto.createHash` (Node API) in the browser and passes `file.arrayBuffer()` (a Promise) into `hash.update`, which expects a Buffer/Uint8Array/string; this will throw at runtime.
- `hash` is both an imported name (`import { hash } from 'crypto'`) and a local variable in `hashFile`, causing a naming collision and likely runtime failure.
- `addFilesToQueue` mutates state incorrectly by reading `fileQueue` and updating `setFileQueue` in a loop without functional updates, risking lost updates and stale closures.
- `useEffect` processing `fileQueue` slices state but can race/starve when multiple files are added quickly; it also logs instead of persisting metadata to library state, so titles/authors won’t appear in the list.
- No actual metadata extraction for author/title (relies on `file.name` and `file.type`), so acceptance criteria for showing title/author cannot be met.
- Duplicate detection uses MD5 and compares only within `newQueue` (not across existing queue + new files), and `duplicateFiles` state is updated per-file in a loop without functional updates, risking missed duplicates.

Acceptance criteria to approve (if fixed):
- Replace Node-only crypto with a browser-safe file hash (e.g., `window.crypto.subtle.digest`) and await `arrayBuffer()` before hashing.
- Use functional state updates (`setFileQueue(prev => ...)`) and compute duplicates against the full existing queue.
- Persist extracted metadata (title, author, fileType) into library state and render it in LibraryUploader so items appear with title/author within 5s.
- Provide a non-throwing, testable path for metadata extraction (even if stubbed) so the list shows something other than raw file objects.
