# LLM Integration Pattern Documentation

## Overview
The **LLM Integration Pattern** provides a reusable, version‑controlled definition for connecting
large language models (LLMs) to Axentx workflows. By defining the pattern once and versioning it,
workflow developers can reference a specific, immutable implementation across any number of
workflows.

## Pattern Definition File
All pattern definitions live under the `patterns/` directory at the repository root.

```
/opt/axentx/surrogate-1/
├─ patterns/
│  ├─ llm_integration_v1.yaml   # Initial version
│  ├─ llm_integration_v2.yaml   # Updated schema / new defaults
│  └─ …                         # Future versions
```

Each file follows a simple YAML schema:

```yaml
# patterns/llm_integration_vX.yaml
name: llm_integration            # Fixed identifier for the pattern
version: X                       # Integer version, matches the file suffix
description: |
  Brief description of what this version does.
settings:
  model: "gpt-4"
  temperature: 0.7
  max_tokens: 1024
  # … additional LLM‑specific configuration …
```

*The `name` field is constant (`llm_integration`) so that workflows can refer to the pattern
without needing to know the exact version.*

## Versioning Logic
Versioning is **explicit and immutable**:

| Action                     | Command / API                                                                 |
|----------------------------|-------------------------------------------------------------------------------|
| **Create a new version**   | Add a new file `llm_integration_v{N}.yaml` where `N = latest_version + 1`.   |
| **Retrieve a version**     | Read the file `llm_integration_v{N}.yaml` from the `patterns/` directory.    |
| **Update a version**       | **Never** modify an existing file. Create a new version instead.            |
| **List available versions**| `ls patterns/llm_integration_v*.yaml` or query via the internal `PatternAPI`.|

The repository includes a tiny helper script (`scripts/pattern_version.py`) that can be used
in CI/CD pipelines to:

```bash
# Get the latest version number
python scripts/pattern_version.py latest llm_integration
# Retrieve the file path for a specific version
python scripts/pattern_version.py path llm_integration 2
```

*(The script is part of the codebase; see `scripts/pattern_version.py` for implementation.)*

## Referencing a Pattern in Workflows
Workflows reference a pattern by **identifier** (`<name>@<version>`). The identifier is parsed
by the workflow engine and resolves to the concrete YAML file.

```yaml
# Example workflow snippet (workflow.yaml)
steps:
  - name: LLM Call
    uses: axentx/llm-step@v1
    with:
      pattern: "llm_integration@2"   # <-- references version 2 of the pattern
```

When the workflow runs:

1. The engine extracts `name = llm_integration` and `version = 2`.
2. It loads `patterns/llm_integration_v2.yaml`.
3. The settings from the pattern are merged with any step‑level overrides.

### Benefits
* **Reusability** – One definition can be shared across dozens of workflows.
* **Traceability** – Every workflow run records the exact pattern version used.
* **Safety** – Immutable version files prevent accidental breaking changes.

## Adding a New Pattern Version (Step‑by‑Step)

1. **Copy the latest version**  
   ```bash
   cp patterns/llm_integration_v2.yaml patterns/llm_integration_v3.yaml
   ```
2. **Edit the new file** – bump the `version` field and adjust any settings.
3. **Commit & push** – the CI pipeline will automatically validate the YAML schema.
4. **Update workflows** (optional) – change the identifier to `llm_integration@3`.
