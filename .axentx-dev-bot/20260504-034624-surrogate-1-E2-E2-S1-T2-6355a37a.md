## 1. Acceptance Criteria  
- **AC1 – Priority Mapping**: The system must map each vulnerability severity level (`critical`, `high`, `medium`, `low`, `info`) to a numeric priority (`1`–`5`, where `1` is highest).  
- **AC2 – Config File Validation**: The `.alert-config.json` file must contain a valid JSON object with a `severityPriority` map; missing or malformed entries should trigger a clear error.  
- **AC3 – Email Payload**: Every email alert must include the vulnerability’s severity and its mapped priority in the subject line.  
- **AC4 – Ordering of Alerts**: When multiple alerts are queued, they must be dispatched in ascending priority order (i.e., `critical` before `high`, etc.).  
- **AC5 – Default Fallback**: If a severity level is not defined in the config, the system defaults to priority `5` (lowest).  
- **AC6 – Logging**: Each priority assignment and any fallback usage must be logged with a timestamp and severity label.  
- **AC7 – Performance**: Priority resolution for a batch of 1000 alerts must complete in < 200 ms on a standard dev machine.
