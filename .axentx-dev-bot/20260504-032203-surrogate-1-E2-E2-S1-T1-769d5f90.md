| # | Requirement | Why it matters |

| 2 | **Three metric types** – `gauge`, `count`, `distribution` | Covers the most common monitoring primitives. |
| 3 | **Sequential timestamps inside a 1‑minute window** (monotonic, never decreasing) | Makes the stream realistic and easy to reason about. |
| 4 | **Deterministic‑ish but varied names** – `metric.<prefix>.<suffix>` (JS) or `<prefix>_<suffix>_<id>` (Python) | Human‑readable, avoids collisions, and mirrors real‑world naming conventions. |

### 2️⃣ Resolved Contradictions  

| Conflict | Resolution |
|----------|------------|
| **Name format** – JS uses `metric.<prefix>.<suffix>` while Python uses `<prefix>_<suffix>_<id>` | Keep both formats because each language ecosystem prefers a different style. The *semantic* parts (prefix + suffix) are identical, and the extra random id in Python guarantees uniqueness when many metrics are generated quickly. |
| **Distribution value** – JS returns an **array of numbers**, Python originally returned a **single number**. | Adopt the richer **array/list** representation for both languages (5‑10 samples). This matches the original JS intent and is more useful for downstream statistical checks. |
| **Timestamp generation** – JS creates a **regularly spaced** series then shuffles, Python picks a random offset per metric. | Use the **regular spacing** approach (deterministic interval) because it guarantees monotonicity without extra sorting. After generation we *optionally* jitter each timestamp by ± 5 ms to avoid a perfectly uniform series while still staying inside the 1‑minute window. |
| **Export style** – JS uses `module.exports`, Python uses a script block. | Keep the idiomatic export for each language (`module.exports` for Node, plain function for Python) and also provide a tiny CLI entry‑point for quick manual runs. |
