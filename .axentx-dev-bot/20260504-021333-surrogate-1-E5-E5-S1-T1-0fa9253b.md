## 2️⃣ Clone the Repository  

```bash
git clone https://github.com/axentx/surrogate-1.git
cd surrogate-1

## 3️⃣ Choose Your Environment Strategy  

You have **three** supported ways to get a working Python environment:

| Strategy | When to use | Commands |
|----------|-------------|----------|
| **Poetry (recommended)** | You want automatic virtual‑env handling, lock‑file reproducibility, and easy script entry points. | See §3.1 |
| **Plain venv + pip** | Poetry cannot be installed (e.g., corporate policies) or you prefer the classic approach. | See §3.2 |
| **Docker** | You need complete isolation, want to avoid installing Python/Poetry locally, or are deploying to a container host. | See §3.3 |

### 3.1 Poetry‑Based Setup  

1. **Install Poetry (if not already present)**  

   ```bash
   curl -sSL https://install.python-poetry.org | python3 -
   # Add Poetry to PATH if the installer didn't do it automatically:
   export PATH="$HOME/.local/bin:$PATH"
   ```

2. **Create the virtual environment & install dependencies**  

   ```bash
   poetry install          # reads pyproject.toml & poetry.lock
   ```

   *Poetry automatically creates a dedicated venv (you can see its location with `poetry env info`).*

3. **Activate the environment (optional, for interactive work)**  

   ```bash
   poetry shell
   ```

### 3.2 Classic venv + pip  

```bash
python3 -m venv .venv               # creates .venv in the repo root
source .venv/bin/activate           # macOS/Linux
# .venv\Scripts\activate            # Windows PowerShell
pip install -r requirements.txt
```

> **Note:** The `requirements.txt` file is kept in sync with the Poetry lock‑file for this fallback path.

### 3.3 Docker (container‑first approach)  

1. **Build the image**  

   ```bash
   docker build -t surrogate-1:latest .
   ```

2. **Run the container** (you’ll need an `.env` file – see §4)  

   ```bash
   docker run --rm \
       --env-file .env \
       -v "$(pwd)/data:/app/data" \   # optional: mount a host data folder
       surrogate-1:latest
   ```

   *If you prefer `docker‑compose`, the repository already ships a `docker-compose.yml` that does the same:*

   ```bash
   docker-compose up --build
   ```

---

## 4️⃣ Configuration – Environment Variables  

Surrogate‑1 relies on a handful of secrets and runtime options. Store them in a **`.env`** file at the project root (never commit this file).

```dotenv
