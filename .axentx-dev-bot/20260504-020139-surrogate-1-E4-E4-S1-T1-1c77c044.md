## 1.  What the current code does

| Layer | What it does | Strengths | Weaknesses |
|-------|--------------|-----------|------------|
| **Discovery** | mDNS service that discovers peers on the local network | Zero‑configuration, works out‑of‑the‑box | No support for remote peers or NAT traversal |
| **Transport** | libp2p stream `/axentx/sync/1.0.0` | Uses libp2p’s secure, multiplexed streams | No back‑pressure handling, no retry logic |
| **Data model** | `Item{Key, Value, Timestamp}` | Simple CRDT‑style “last‑write‑wins” | No vector clocks, no versioning, no conflict‑resolution hooks |
| **Sync logic** | Push‑only on connect + bidirectional reply | Keeps both sides in sync | Full‑store push on every update → O(n) traffic |
| **Status** | HTTP endpoint `/sync/status` | Easy to monitor | Exposes internal state; no authentication |
| **API** | `SetItem`, `GetItem`, `GetStatus` | Very small surface | No batch API, no delete semantics |

---

## 2.  What we want to keep

1. **Zero‑config discovery** – mDNS is great for local‑area networks.
2. **libp2p stream abstraction** – gives us encryption, multiplexing, and a clean API.
3. **Last‑write‑wins** – simple conflict resolution that works for most key/value stores.
4. **HTTP status endpoint** – useful for observability and debugging.

---

## 3.  What we should change

| Area | Problem | Fix |
|------|---------|-----|
| **Discovery** | Only local peers | Add optional *gossipsub* or *MDNS+Bootstrap* so the node can join a larger network. |
| **Transport** | No back‑pressure, no retries | Wrap the stream in a `bufio.Reader/Writer` with a configurable buffer size; retry on transient errors. |
| **Data model** | Full‑store push on every change | Switch to *incremental sync*: send only the changed `Item` (or a list of changed keys) and let the peer merge. |
| **Sync logic** | O(n) traffic on every update | Keep a *change‑log* (append‑only slice of `Item`) and send only the tail of the log. |
| **Conflict resolution** | Only last‑write‑wins, no user hooks | Expose a `ConflictResolver` interface that callers can plug in. |
| **Status API** | Unauthenticated, leaks internal state | Add basic auth or an API key; expose only the fields that are safe for external consumption. |
| **API** | No delete, no batch | Add `DeleteItem(key)` and `SetItems([]Item)` for bulk operations. |
| **Testing** | Only a single test stub | Add integration tests that spin up two nodes, perform concurrent writes, and assert eventual consistency. |
