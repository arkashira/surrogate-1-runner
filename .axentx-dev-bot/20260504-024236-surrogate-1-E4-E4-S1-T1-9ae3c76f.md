## Unified, Production‑Ready Metrics Stack  

Below is a **single, coherent design** that merges the strongest ideas from the two proposals:

| Aspect | What we keep from **Proposal 1 (Python)** | What we keep from **Proposal 2 (Go)** | How we combine them |
|--------|--------------------------------------------|----------------------------------------|----------------------|
| **Metric collection** | Easy‑to‑read `psutil`‑based CPU / memory snapshot, JSON‑line log file, automatic 30‑day retention, unit‑tests. | Strongly typed `MetricRecord` struct, in‑memory ring buffer, thread‑safe `sync.RWMutex`. | Implement the collector **in Go** (fast, compiled, safe) and **reuse the same JSON‑line format** that the Python side already expects. |
| **HTTP API** | Flask `/metrics` endpoint that returns a JSON list. | Go `http.Handler` that streams the JSON‑line file or the in‑memory slice. | Expose a **single Go HTTP server** that serves both `/metrics` (JSON) and `/dashboard` (HTML) – no need for two Flask apps. |
| **Dashboard** | Simple Jinja2 template that renders a table. | None (focus on backend). | Keep the **HTML/Jinja2 template** but render it from Go using the standard `html/template` package (identical markup, no Python runtime). |
| **Testing** | `unittest` for collector, Flask endpoint, dashboard. | None (but Go’s `testing` is idiomatic). | Provide **both Python‑style integration tests** (via `requests`) and **Go unit tests** for the core collector. |
| **Retention / cleanup** | `cleanup_old_metrics()` that rewrites the log file. | Not present. | Use the Go cleanup routine (runs daily via a goroutine) that truncates the JSON‑line file to the last 30 days, keeping the same semantics. |
| **Packaging / deployment** | Plain source tree, run with `python app.py`. | Go module, compiled binary. | Deliver **one compiled binary** (`metrics-server`) plus the static `templates/` directory. No Python runtime required in production, but developers can still run the existing Python unit‑tests locally if they wish. |
