## 1. Acceptance Criteria
- **AC1** – Every `/ai` request results in a single audit entry containing **exactly** the fields: `timestamp` (ISO‑8601), `user_id` (string), `model` (string), `request_size_bytes` (int), and `response_status` (enum: `success`, `error`, `rate_limited`).  
- **AC2** – Audit entries are persisted to a **JSON Lines** file (`audit-YYYY-MM-DD.jsonl`) located in the configured `audit_log_dir`. Each line must be a valid JSON object matching the schema from AC1.  
- **AC3** – The CLI command `audit-cli query --from <date> --to <date> [--user <id>] [--model <name>]` returns all matching entries in chronological order, formatted as pretty‑printed JSON.  
- **AC4** – Log rotation runs automatically at **00:00 UTC** each day, closing the current file and creating a new file for the new day. The previous day’s file is moved to an `archive/` sub‑directory and compressed (`.gz`).  
- **AC5** – The system must **not lose** any entries during rotation: every request made before rotation appears in the previous day’s file, and every request after rotation appears in the new file.  
- **AC6** – If the `audit_log_dir` is missing or unwritable, the request handling pipeline returns an error with code `500` and logs a warning to the application logger.  
- **AC7** – The audit log file size never exceeds **100 MB** per day; if the size limit is reached before midnight, a new file with an incremented suffix (`audit-YYYY-MM-DD-1.jsonl`, `audit-YYYY-MM-DD-2.jsonl`, …) is created and subsequently archived with the same daily rotation policy.
