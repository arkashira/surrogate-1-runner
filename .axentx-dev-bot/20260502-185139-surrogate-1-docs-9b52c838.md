## Latency metrics

The system captures end-to-end pipeline latency and exposes a bounded histogram (default: 10,000 samples). Oldest samples are evicted when capacity is exceeded.

- Record latency with `record_latency_ms(ms)` (rejects negative values with `ValueError`).
- Query statistics: `mean()`, `min()`, `max()`, `count()`, and `quantile(p)` (linear interpolation; returns `None` when no samples).
- Health endpoint: `GET /health/latency` returns 200 with:
  - No samples: `{ ok: false, status: "no_samples", message: "no latency samples available" }`
  - With samples: `{ p50, p95, p99, mean, min, max, slo_breach }` where `slo_breach === true` iff `p95 >= 1000` (ms).
- Timing is captured in a `finally` block (e.g., `time.perf_counter()`) so latency is recorded even when the pipeline raises.
- Concurrent writes are safe and preserve data integrity under bounded storage.
