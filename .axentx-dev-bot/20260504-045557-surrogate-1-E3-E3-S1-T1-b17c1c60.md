## Unified Design – “AxentX Surrogate 1”  
A production‑ready, low‑latency **asynchronous signature‑comparison service** that can be dropped into any Node‑or‑Go‑based backend.  

The design merges the strongest ideas from the two proposals:

| Aspect | What works best | How we combine it |
|--------|----------------|-------------------|
| **Concurrency model** | Go worker‑pool (fixed‑size, cheap goroutine scheduling) | Keep the Go pool **as the canonical executor** – it owns the job queue, does the heavy work, and calls the fast‑path signature service. |
| **Fast‑path signature check** | JavaScript batch‑queue that groups requests and talks to a local HTTP endpoint | Run a **single Node.js process** that exposes `/internal/compare-signatures`. It receives batches, does the cheap comparison, and returns an array of booleans. |
| **Batching strategy** | Configurable `batchSize` + `maxDelayMs` that guarantees < 50 ms 99‑pct latency | The Node queue implements exactly the `AsyncSignatureQueue` class from Candidate 2. The Go side simply POSTs a *single‑item* batch; the Node side will automatically coalesce many concurrent POSTs into the same internal batch. |
| **Resource limits** | 16‑worker Go pool (Candidate 1) + back‑pressure on the channel (Candidate 2) | Use a **configurable pool size** (default 16). The job channel is buffered (`workerCount*2`). If the buffer is full, the submit call blocks briefly – this prevents silent drops while still keeping the system responsive. |
| **Observability** | Simple latency warning in Go worker (Candidate 2) | Keep the per‑job latency check and add **Prometheus metrics** (optional) for queue depth, batch size, and 99‑pct latency. |
| **Graceful shutdown** | Signal handling in Go (Candidate 2) | Preserve the same `Stop()` logic – on SIGINT/SIGTERM close the quit channel, wait for workers, then ask the Node process to shut down. |
| **Deployment footprint** | Two binaries (Go + Node) that can be started by a single systemd unit or Docker‑compose file | Provide a small wrapper script that launches the Node process in the background, then execs the Go binary. The wrapper forwards signals to both processes. |

## 1. Architecture Overview  

```
+-------------------+          HTTP POST (single item)          +-------------------+
|   Go Worker Pool  |  <--------------------------------------  |   Node.js Queue   |
| (goroutine pool) |                                          | (AsyncSignatureQ) |
+-------------------+          HTTP POST (batch) -------------->+-------------------+
        |  ^                                                    |
        |  |  Job submission (Job{id,signature, …})            |
        v  |                                                    |
+-------------------+          internal channel               |
|   HTTP Frontend   |  --------------------------------------> |
| (optional)        |  (receives external requests)            |
+-------------------+                                           
```

* **Go** is the *engine* – it pulls jobs from a channel, runs the cheap signature check, and only does the expensive downstream work if the signature is new.  
* **Node** is the *signature‑service* – it batches many tiny POSTs into a single internal batch, runs the comparison logic once per batch, and returns a boolean for each request.  
* The **batching** guarantees that even if a single request arrives, the latency added by the extra HTTP hop stays well under the 50 ms target because the Node side will flush after `maxDelayMs` (default 20 ms) or when the batch fills (default 48).
