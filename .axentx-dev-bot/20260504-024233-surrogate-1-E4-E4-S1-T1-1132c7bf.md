## Unified “Drift‑Detection‑as‑Code” Solution  

Below is a **single, production‑ready** GitHub‑Actions workflow together with the minimal Python helper scripts you need to run a drift‑detection tool on every pull‑request (and on manual dispatch).  

It combines the strongest ideas from the two drafts you posted:

| What we keep | Why it’s useful |
|--------------|-----------------|
| **Trigger on PR opened / synchronize / reopened** (and `workflow_dispatch`) | Guarantees every code change is checked, while still allowing ad‑hoc runs. |
| **Explicit `permissions` (`pull‑requests: write`)** | Needed for the `github-script` step that posts a comment. |
| **Use the latest stable actions (`checkout@v4`, `setup‑python@v5`, `github‑script@v7`)** | Improves security and gets the newest bug‑fixes. |
| **Conditional install of `requirements.txt` *or* a `pyproject.toml`** | Works for both classic pip‑based projects and modern Poetry/PEP‑517 builds. |
| **A tiny wrapper script (`bin/drift‑detector.py`) that runs the real detector and writes a JSON report** | Guarantees a **single source of truth** for the exit‑code and the report file, avoiding the infinite‑recursion bug in the original `drift‑detection.py`. |
| **`set-output` / `$GITHUB_OUTPUT` usage** | The modern, recommended way to pass data between steps. |
| **`always()` on the comment step** | Guarantees a comment even when the job fails (so the author sees *why* the check failed). |
| **Rich comment formatting (emoji, collapsible details, raw JSON)** | Makes the PR discussion easy to read for humans and parsable for bots. |
| **Fail the job when drift is detected** | Keeps the PR check red until the drift is resolved. |

It **must exit with a non‑zero status** when drift is found. The wrapper (`drift-detector.py`) handles the plumbing for the CI job.

### 2️⃣ `bin/drift-detector.py` – the stable wrapper  

```python
#!/usr/bin/env python3
"""
drift-detector.py
-----------------
A thin wrapper that:

1. Executes the real detector (bin/drift-detection.py) with the requested environment.
2. Captures its JSON output.
3. Writes that output to `drift_report.json` (so the GitHub‑script step can read it).
4. Exits with the same return‑code as the detector (0 = no drift, >0 = drift).
"""

import argparse
import json
import subprocess
import sys
from pathlib import Path

REPORT_PATH = Path("drift_report.json")


def run_detector(env: str) -> int:
    """Run the real detector and persist its JSON output."""
    # NOTE: we call the *real* detector, not this wrapper – avoid recursion!
    cmd = [sys.executable, "bin/drift-detection.py", "--env", env]

    # Capture stdout (the JSON report) and stderr (debug info)
    result = subprocess.run(
        cmd, capture_output=True, text=True, check=False
    )

    # Write whatever stdout we got – even if it is empty – to a file for later steps.
    try:
        # Validate that stdout is valid JSON; if not, fall back to raw text.
        json_obj = json.loads(result.stdout) if result.stdout.strip() else {}
        REPORT_PATH.write_text(json.dumps(json_obj, indent=2))
    except json.JSONDecodeError:
        # Keep the raw output; the comment step will still show it.
        REPORT_PATH.write_text(result.stdout)

    # Propagate the detector's exit status.
    return result.returncode
