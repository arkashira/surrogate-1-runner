## Pipeline basics

- Stages are executed in order. Each stage receives interpolated inputs and its outputs are merged into the shared context under a **stable stage key** (`stage.name` or `type_index`).
- Inputs are interpolated before execution: `{{path.to.value}}` tokens are resolved recursively against the current context. Strings have tokens replaced (undefined → empty string), while arrays and objects preserve structure and types.
- Top-level convenience merging only happens when keys do not already exist in context.
- Signal-type stages validate inputs and throw `PipelineAbortError` (machine-readable `PipelineError` code and message that includes the stage name and failing value). The pipeline aborts immediately on validation or interpolation failures.
- Signal generator task accepts inputs (e.g., symbol, horizon) and returns `{ signal_id, side, price, confidence }` with required fields present and typed. Missing or invalid signal input aborts with a clear, machine-readable error code and message referencing the stage name and failing value.
- Runners expose `stageNames` (deterministic keys) so callers can inspect stage ordering and locate outputs in context.
