## 1. Acceptance Criteria (measurable)

- **AC1 – Audit Trail Creation**  
  When a Helm rollback is initiated, a new record is inserted into the `helm_rollback_audit` table with non‑null `action_id`, `attempted_at`, and `status` set to `"started"`.

- **AC2 – Audit Trail Update**  
  Upon completion (success or failure), the same audit record is updated with `completed_at`, `status` (`"succeeded"` or `"failed"`), and `output` containing Helm stdout/stderr.

- **AC3 – Correct UPDATE Target**  
  The UPDATE statement uses the `id` returned from the INSERT, not a fragile timestamp filter.

- **AC4 – Context Cancellation Handling**  
  If the rollback context is cancelled, the Helm process is killed, the function returns immediately with the cancellation error, and no audit record is updated beyond the initial insert.

- **AC5 – Type Consistency**  
  `RollbackConfig.TargetRevision` (string) is correctly converted to an integer for the Helm command and for database insertion.

- **AC6 – Truncated Log Statement Fixed**  
  The logger call after the UPDATE has a complete message string and logs the error when the UPDATE fails.

- **AC7 – Secret Handling**  
  Helm stderr is stored in the audit record only if it does not contain any sensitive tokens (e.g., `AWS_SECRET_ACCESS_KEY`). A sanitization step removes such tokens before storage.

- **AC8 – Unit Test for Nil DB**  
  Calling `RollbackHelm` with `DB: nil` does not panic and returns a non‑nil error indicating the missing database connection.
