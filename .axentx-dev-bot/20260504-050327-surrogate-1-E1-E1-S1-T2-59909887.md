## Unified, production‑ready notification configuration

Below is a single, self‑contained set of files that pulls together the strongest ideas from both candidates:

* **`application.properties`** – a single, strongly‑typed configuration that supports Slack, Email, and generic Webhook channels, plus global alert settings.
* **`logback.xml`** – a Logback configuration that routes all *INFO*‑level (and higher) logs to the console, to a Slack appender, and to an optional email appender.  
  The Slack appender is lightweight (uses the `logback‑slack‑appender` library) and the email appender uses the standard `SMTPAppender`.

> **Why this works**  
> * **Strong typing** – the `notification.channel.type` property guarantees that only one channel is active at a time, preventing accidental double‑sending.  
> * **Extensibility** – the generic webhook block lets you plug in any HTTP endpoint without touching code.  
> * **Fail‑fast** – if a required property is missing, the application will fail to start, making mis‑configuration obvious.  
> * **Actionable** – the Logback configuration is ready‑to‑use; just drop the two JARs into your classpath.

---

### 1. `application.properties`

```properties
# ------------------------------------------------------------------
# 1. Notification channel selection
# ------------------------------------------------------------------
# Allowed values: slack, email, webhook
notification.channel.type=slack

# ------------------------------------------------------------------
# 2. Slack configuration (used only when channel.type=slack)
# ------------------------------------------------------------------
notification.slack.webhook-url=<slack-webhook>
notification.slack.channel=#alerts
notification.slack.username=SurrogateBot
# Optional: set a custom icon
notification.slack.icon-url=https://example.com/icon.png

# ------------------------------------------------------------------
# 3. Email configuration (used only when channel.type=email)
# ------------------------------------------------------------------
notification.email.smtp-host=smtp.example.com
notification.email.smtp-port=587
notification.email.smtp-username=alerts@example.com
notification.email.smtp-password=supersecret
notification.email.from=alerts@example.com
notification.email.to=dev-team@example.com,ops@example.com
# Optional: TLS/SSL
notification.email.smtp-starttls.enable=true
notification.email.smtp-auth=true

# ------------------------------------------------------------------
# 4. Generic webhook configuration (used only when channel.type=webhook)
# ------------------------------------------------------------------
notification.webhook.url=https://example.com/webhook
notification.webhook.method=POST
notification.webhook.headers=Content-Type:application/json,Authorization:Bearer XYZ
# Optional: timeout in ms
notification.webhook.timeout=5000

# ------------------------------------------------------------------
# 5. Global alert settings
# ------------------------------------------------------------------
# Enable or disable alerts entirely
notification.alert.enabled=true
# Number of drift events that must occur before an alert is sent
notification.alert.threshold=1
# Minimum log level that triggers an alert (INFO, WARN, ERROR)
notification.alert.level=INFO
```

> **Tip**: Store secrets (SMTP password, Slack webhook URL, etc.) in a secrets manager or environment variables and reference them with `${…}` if your framework supports it.
