## Unified On‑boarding System – “One‑Stop‑Shop”  

Below is a **single, coherent design** that merges the strongest ideas from the two proposals you shared:

| What we keep from **Candidate 1** | What we keep from **Candidate 2** | Why it matters |
|-----------------------------------|-----------------------------------|----------------|
| **Markdown‑based documentation** (central `docs/onboarding.md`) | **CLI‑driven interactive training** (`interactive_training()`) | New hires can read the guide *and* walk through it step‑by‑step in the terminal. |
| **`OnboardingManager` class** that loads a JSON config, tracks progress, stores feedback | **Simple JSON persistence** (`progress.json`) and per‑user step flags | A clean OO API **plus** a lightweight, human‑readable persistence format. |
| **Unit‑test suite** (`tests/onboarding/…`) | **Explicit error handling** (FileNotFoundError, path handling) | Guarantees correctness and gives clear diagnostics when files are missing. |
| **Progress as a float** (0‑1) | **Per‑user, per‑module boolean** | We support **both**: a global “module completeness” (percentage) **and** a per‑user “has this step been done?” flag. |
| **Feedback collection** | **Progress report generation** | Gives mentors data to act on and lets the system summarise a newcomer’s status. |

The result is a **single Python package** that:

1. **Loads** onboarding material from a Markdown file.  
2. **Exposes** a clean OO API (`OnboardingManager`) for programmatic use.  
3. **Provides** a tiny CLI (`python -m onboarding.cli …`) for interactive sessions.  
4. **Persists** data in two JSON files (`config/onboarding_config.json` & `data/progress.json`).  
5. **Ships** with documentation, a FAQ, and a full test suite.
