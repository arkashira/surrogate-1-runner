## 1. Acceptance Criteria  
*(All criteria are measurable and testable)*  

- **AC1 – Event Routing**  
  The webhook handler must correctly route at least the following Stripe events to their dedicated processors:  
  - `invoice.payment_succeeded`  
  - `invoice.payment_failed`  
  - `customer.subscription.updated`  
  - `customer.subscription.deleted`  

- **AC2 – Signature Verification**  
  The handler must reject any request that does not contain a valid Stripe signature header (`Stripe-Signature`) or whose signature does not match the payload using the configured webhook secret.  
  - Response: `401 Unauthorized` with JSON body `{ "error": "Invalid signature" }`.

- **AC3 – Idempotency**  
  Duplicate events (same `id`) must be ignored after the first successful processing.  
  - Response to duplicate: `200 OK` with body `{ "status": "ignored" }`.

- **AC4 – Subscription Status Update**  
  Upon receiving `customer.subscription.updated` or `customer.subscription.deleted`, the handler must update the corresponding `Subscription` record in the database to reflect the new status (`active`, `past_due`, `canceled`).  
  - The update must be persisted before responding `200 OK`.

- **AC5 – Invoice Handling**  
  - `invoice.payment_succeeded`: Mark the invoice as paid and trigger any downstream “usage” or “quota” updates.  
  - `invoice.payment_failed`: Mark the invoice as unpaid, set the subscription to `past_due`, and enqueue a retry job.  
  - Response for both: `200 OK` with body `{ "status": "processed" }`.

- **AC6 – Error Handling**  
  Any unexpected exception must be logged, and the handler must return `500 Internal Server Error` with body `{ "error": "Internal server error" }`.

- **AC7 – Logging**  
  All processed events must be logged with event type, ID, and processing outcome.
